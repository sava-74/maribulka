## 1. Контекст

Проект расширяется: изначально система создавалась только для фотографа, но теперь объединяет фотостудию и парикмахерскую в одном салоне. Расписание зависит от мастеров обоих направлений.

Текущее состояние кода: существует только бинарная проверка `isAdmin` в auth store. Все роли и права — заглушки, нигде не применялись. Начинаем с нуля.

## 2. Роли пользователей

### 2.1 Иерархия

```
вход на сайт
├── admin         (1 в базе)
├── SuperUser     (до 3)
├── AUser         (до 3)
├── ProUser       (до 20)
└── user          (много, на будущее)
    └── без авторизации (только просмотр)
```

### 2.2 Описание ролей

| Роль | Кто | Особенности |
|------|-----|-------------|
| `admin` | Технический администратор сайта | Следит за ресурсом, управляет всеми пользователями, видит все логи. Не обязательно участвует в бизнес-процессе. Может совмещаться с SuperUser. |
| `SuperUser` | Руководитель бизнеса (директор салона) | Полный доступ ко всему. Логи не нужны в ежедневной работе — при необходимости запрашивает у admin. |
| `AUser` | Менеджер/Администратор салона | Работает с клиентами и расписанием. Частичный доступ к справочникам. Финансы не видит. |
| `ProUser` | Исполнитель (фотограф / парикмахер) | Только свои заказы: создать, просмотреть, принять оплату. Чужие заказы видит как "Время занято". |
| `user` | Посетитель сайта | Публичная часть + комментарии. **На будущее.** |

## 3. Специализации

В карточке пользователя (для SuperUser / AUser / ProUser) — три галочки специализации:

- **Администратор** — участвует в административных процессах
- **Фотограф** — появляется в списке исполнителей для фото-заказов
- **Парикмахер** — появляется в списке исполнителей для парикмахерских заказов

**Правила:**
- Специализации независимы — можно отметить любую комбинацию
- SuperUser по умолчанию может работать в любой специализации
- Если галочка специализации не стоит — имя мастера НЕ появляется в выпадающем списке при создании заказа соответствующего типа
- Хранится в таблице `users` как отдельные boolean поля: `is_admin_role`, `is_photographer`, `is_hairdresser`

## 5. Управление пользователями

| Роль | Кто может создавать/редактировать |
|------|----------------------------------|
| `admin` | Любого пользователя любой роли |
| `SuperUser` | AUser, ProUser и user |
| `AUser` | **Нет прав на управление пользователями** |
| `ProUser` | Нет прав |
| `user` | Нет прав |

**Самостоятельная регистрация:** неавторизованный пользователь может зарегистрироваться → получает роль `user`.

## 6. Гранулярные права (система 1С-подобная)

### Принцип
Роль задаёт **права по умолчанию**. Admin может **переопределить любое право** для любого конкретного пользователя — не меняя его роль.

### Структура БД

```sql
-- Таблица пользователей (расширить существующую)
ALTER TABLE users ADD COLUMN role ENUM('admin','superuser','auser','prouser','user');
ALTER TABLE users ADD COLUMN is_photographer BOOLEAN DEFAULT FALSE;
ALTER TABLE users ADD COLUMN is_hairdresser BOOLEAN DEFAULT FALSE;
ALTER TABLE users ADD COLUMN is_admin_role BOOLEAN DEFAULT FALSE;
ALTER TABLE users ADD COLUMN name VARCHAR(255);
ALTER TABLE users ADD COLUMN created_at DATETIME DEFAULT NOW();

-- Таблица индивидуальных прав (переопределение defaults)
CREATE TABLE user_permissions (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  section VARCHAR(64) NOT NULL,   -- 'calendar', 'bookings', 'income', 'clients', etc.
  action VARCHAR(32) NOT NULL,    -- 'view', 'create', 'edit', 'delete'
  allowed BOOLEAN NOT NULL,
  granted_by INT,                 -- кто установил переопределение (FK на users.id)
  created_at DATETIME DEFAULT NOW(),
  UNIQUE KEY uq_user_section_action (user_id, section, action),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (granted_by) REFERENCES users(id) ON DELETE SET NULL
);
```

### Алгоритм проверки прав (frontend + backend)

```
hasPermission(user, section, action):
  1. Проверить таблицу user_permissions для этого user_id + section + action
  2. Если запись есть → вернуть её значение (индивидуальное переопределение)
  3. Если записи нет → вернуть default для роли пользователя
```

### Загрузка прав — стратегия

Индивидуальные права загружаются **один раз при логине** и кладутся в сессию на backend + в store на frontend. При изменении прав admin'ом — изменения применяются только после следующего логина пользователя (или принудительного сброса сессии).

Это упрощает реализацию и исключает N+1 запросы к БД на каждый endpoint.

### Дефолтные права по ролям

При создании пользователя автоматически НЕ создаются записи в `user_permissions` — дефолты читаются из конфига ролей в коде. Записи в `user_permissions` создаются ТОЛЬКО когда admin меняет что-то индивидуально.

### Структура ответа auth.php?action=check (расширенная)

```json
{
  "success": true,
  "isAuthenticated": true,
  "user": {
    "id": 5,
    "login": "ivan",
    "name": "Иван",
    "role": "prouser",
    "specializations": {
      "photographer": true,
      "hairdresser": false,
      "admin_role": false
    },
    "permissions": [
      { "section": "clients", "action": "view", "allowed": true }
    ]
  }
}
```

`permissions` — только индивидуальные переопределения. Если массив пуст — используются дефолты роли.

## 7. Журналы логирования

**Два типа журналов:**

1. **Журнал действий** (`action_log`) — кто/когда/что сделал в системе. Есть у всех авторизованных ролей.
2. **Журнал пользователей** (`user_log`) — операции с аккаунтами (создание, изменение прав, вход/выход). Доступен только admin.

**Просмотр журналов:**
- `admin` — видит все журналы
- `SuperUser` / `AUser` / `ProUser` — не видят журналы (при необходимости запрашивают у admin)

- **Кнопка "Добавить"** — активна, создаёт новый заказ

## 9. Архитектура реализации

### Подход
**Роли как шаблон прав по умолчанию + таблица индивидуальных переопределений.**

Единая функция проверки `hasPermission(section, action)` — сначала смотрит на индивидуальные права, потом на дефолты роли.

### Frontend
- `auth.ts` store: добавить `userRole`, `userSpecializations`, функцию `hasPermission()`
- Лаунч: рендерить пункты по `hasPermission()`
- Кнопки действий: скрывать/блокировать по правам
- Календарь: фильтровать чужие записи для ProUser

### Backend
- `auth.php`: возвращать роль + специализации + индивидуальные права при check/login
- Каждый API endpoint: проверять права через общую функцию `checkPermission()`
- Новый endpoint: `api/users.php` — CRUD пользователей
- Новый endpoint: `api/permissions.php` — управление индивидуальными правами

### Секции для системы прав

```
calendar          — Календарь
bookings          — Записи (таблица)
income            — Доход
expenses          — Расход
reports           — Отчёты
clients           — Клиенты
shooting_types    — Типы съёмок
promotions        — Акции
expense_categories — Категория расходов
settings          — Настройки
home              — Домашняя страница
news              — Новости
portfolio         — Портфолио
users             — Управление пользователями
logs              — Журналы
```

## 10. UI — Раздел "Пользователи" (Справочники)

### 10.1 Таблица пользователей

Доступна: **admin** и **SuperUser**.

**Колонки таблицы:**

| Колонка | Описание |
| ------- | -------- |
| ФИО | Полное имя |
| Логин | Логин для входа |
| Тип | Роль: admin / SuperUser / AUser / ProUser |
| Админ | Галочка специализации |
| Фотограф | Галочка специализации |
| Парикмахер | Галочка специализации |
| Тип зарплаты | Оклад / Процент / Оклад + % |
| Дата приёма | Дата приёма на работу |
| Примечания | Произвольный текст |

**Уволенные сотрудники** в основной таблице не отображаются (фильтруются по `fired_at IS NULL`). Удалить пользователя нельзя — он участвует в исторических документах (заказы, финансы). Уволенный не учитывается ни в каких бизнес-процессах: не появляется в списках исполнителей, не может войти в систему.

### 10.2 БД — расширение таблицы users

```sql
ALTER TABLE users ADD COLUMN full_name VARCHAR(255);
ALTER TABLE users ADD COLUMN hired_at DATE;
ALTER TABLE users ADD COLUMN fired_at DATE;          -- NULL = работает, дата = уволен
ALTER TABLE users ADD COLUMN salary_type ENUM('fixed','percent','fixed_percent');
ALTER TABLE users ADD COLUMN notes TEXT;
```

### 10.3 Меню действий (клик на строку таблицы)

**Для admin:**

- Просмотр
- Редактировать
- Изменить права *(только admin)*
- Уволить

**Для SuperUser:**

- Просмотр
- Редактировать
- Уволить

### 10.4 Модалка создания / редактирования пользователя

**Поля:**

- ФИО
- Логин
- Пароль (при создании обязательно, при редактировании — необязательно, менять отдельно)
- Тип (роль): выпадающий список
- Специализации: три галочки — Администратор, Фотограф, Парикмахер
- Тип зарплаты: выпадающий список (Оклад / Процент / Оклад + %)
- Дата приёма на работу
- Примечания
- Галочка **"Уволен"** — при установке автоматически проставляется текущая дата в `fired_at`. **Необратимое действие** — восстановления нет. Повторный приём оформляется как новая запись.

### 10.5 Модалка "Изменить права" (только admin)

Открывается из меню действий. Показывает матрицу индивидуальных переопределений для конкретного пользователя:

- Список всех секций (calendar, bookings, income, expenses, clients, ...)
- Для каждой секции: действия (view / create / edit / delete)
- Текущее значение: **По умолчанию** (серый) / **Разрешено** (зелёный) / **Запрещено** (красный)
- Admin может переключить любое значение — оно сохраняется в `user_permissions`
- Кнопка "Сбросить всё" — удаляет все индивидуальные переопределения для пользователя

### 10.6 Простые user (посетители сайта)

Хранятся в **отдельной таблице** — не смешиваются с сотрудниками. **На будущее.**
