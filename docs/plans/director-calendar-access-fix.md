# Исправление: Директор не видит записи в календаре

**Дата создания:** 3 апреля 2026  
**Приоритет:** Критический (P1)  
**Статус:** На утверждение

---

## 📋 Анализ задачи

**Проблема:** Пользователи с ролью "Директор" не видят записи в календаре записей.

**Симптомы:**
- Календарь отображается пустым
- Данные о записях не загружаются
- Ошибки авторизации/сессии в консоли или логах API

---

## 🔴 Root Cause Analysis

### Причина 1: Некорректные настройки session cookie для localhost
**Файл:** `api/session.php`

Проблема: параметр `secure: true` в `session_set_cookie_params()` ломает сессии при работе через HTTP (localhost), так как флаг `secure` требует HTTPS.

### Причина 2: Отсутствие проверки авторизации в bookings.php
**Файл:** `api/bookings.php`

Проблема: endpoint не проверяет сессию пользователя, отдаёт данные всем запросам без авторизации. Это нарушает безопасность и может вызывать проблемы с доступом для авторизованных пользователей.

---

- [ ] Сессии корректно работают на localhost (HTTP) и на продакшене (HTTPS)

- [ ] Директор видит все записи в календаре
- [ ] Безопасность API сохранена (нет утечки данных)

## 📝 План исправления

### Priority 1: Исправить session cookie для localhost

**Файл:** `api/session.php`

**Текущая проблема:**
```php
session_set_cookie_params([
    'secure' => true,  // ❌ Ломает сессии на HTTP (localhost)
    'httponly' => true,
    'samesite' => 'None'
]);
```

**Решение:**
```php
function initSession() {
    if (session_status() === PHP_SESSION_NONE) {
        $isLocalhost = isset($_SERVER['HTTP_HOST']) && 
                       strpos($_SERVER['HTTP_HOST'], 'localhost') !== false;
        
        session_set_cookie_params([
            'lifetime' => 0,
            'path' => '/',
            'domain' => '',
            'secure' => !$isLocalhost,        // true только для HTTPS
            'httponly' => true,
            'samesite' => $isLocalhost ? 'Lax' : 'None'
        ]);
        session_start();
    }
}
```

**Обоснование:**
- `secure: false` для localhost — позволяет сессиям работать по HTTP
- `samesite: Lax` для localhost — корректная работа в браузере без HTTPS
- `secure: true` для продакшена — безопасность cookie при передаче

---

### Priority 2: Добавить проверку сессии в bookings.php

**Файл:** `api/bookings.php`

**Изменения:** добавить после `<?php` и `header()`, перед подключением `database.php`:

```php
<?php
header('Content-Type: application/json');
header('Access-Control-Allow-Origin: http://localhost:5173');
header('Access-Control-Allow-Credentials: true');
header('Access-Control-Allow-Methods: GET, POST, OPTIONS');
header('Access-Control-Allow-Headers: Content-Type');

if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') {
    http_response_code(200);
    exit;
}

// ✅ Добавить:
require_once 'session.php';
initSession();

// Проверка авторизации
if (!isset($_SESSION['user'])) {
    http_response_code(401);
    echo json_encode(['error' => 'Требуется авторизация']);
    exit;
}

// Далее существующий код...
require_once 'database.php';
```

**Обоснование:**
- Защита от несанкционированного доступа к данным
- Возврат 401 для неавторизованных запросов
- Явное сообщение об ошибке
